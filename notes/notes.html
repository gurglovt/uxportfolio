UX CASE STUDY

Notes App

Creating a notes app that allows users to record their ideas as quickly as possible.

This is a personal project. I was not hired by anyone to design a notes app.

Empathise

Key Findings

Users use whatever is at hand and convenient

They will use solutions that are impractical in the long term if they at least work in the instant that they are needed to.

A notes app should do both: work now, and not at the expense of organisation later.
A notes app should be at hand and it should be quick to make notes and set reminders.
Offline support should be included. If there's a multi-device transfer feature, it should auto-sync when a connection is established.
Users tend towards solutions that serve multiple functions

E.g. a notes app that allows the user to transfer media between devices, or a calendar app with built-in reminders.

Organisation should be intuitive and automatic. The user shouldn't need to think so much.
The app should be multi-functional.
Notes are ephemeral

Notes are often edited and deleted. It's useful to have a 'whiteboard' space with immediately important information.

The app should make these easy to do, while allowing easy error recovery.
This could be a note or a dedicated 'whiteboard' section.

Personas

Lily needs an app that collects her ideas cohesively and effortlessly. Note-taking shouldn't be an active task, but one that is ready for her whenever her brain conjures something up. She needs a net that catches and catalogues her ideas as they appear.

Alan needs an app that can hold onto all the things he wants to remember, for business and for pleasure. He needs something that's ready for him whenever he needs it without complication. He needs impermanent lists and reminders, but also permanent study notes.

Define

Initial Ideas

Feature List

Architecture

A Notepads section and a Lists/Reminders section with footer navigation. Notepads have notes and subnotes, lists are just lists.

Block System

Note architecture has a system of "blocks" like Notion. After typing, press the return key to finish the current block and make a new one.

Whiteboard/Scratchpad

A feature that works like a messenger with a keyboard-ready typing zone. Options to wipe, convert to a new note, or append to existing note.

Note Browser

A system that looks similar to a file browser. Expand/collapse all notebooks option, recent sort, favourites section.

For this version, the focus is on making a functional, offline, digital notepad. No sign-up/share feature for now.

Test 1

A brief and less rigidly-structured user test to understand how well the initial design is understood.

Aims

  • See if users can understand what the app is and does
  • See if users find the placement and navigation intuitive
  • Identify areas of confusion before making the prototype

Methodology

  1. Preface that these are screenshots and aren't interactable
  2. For each of the four screens, ask:
    • Could you tell me what this is?
    • What do you think you can do here?
  3. What are your overall thoughts when you look at the four screens together?

Key Findings

Confusion around "Dump"

Both users were confused about the Dump's purpose and the arrow that expands it. User 2 stated: "I don't get why it's called 'Dump'!"

Lists vs Notebooks separation unclear

Users questioned why lists and notebooks are separate. User 2 wanted these functions consolidated onto one page.

Expand/contract buttons misunderstood

User 2 misinterpreted the expand/contract all buttons as a full-screen mode.

Navigation concerns

The use of the same UI on both Home and Notebooks screens caused confusion about whether it was global.

Actions Taken

Dump

Rework the Dump's presentation to new users. Make expand icon consistent with iconography used elsewhere.

Lists

Consolidate the List section with the Notebooks section. Add reminder toggle to list item-type blocks.

Navigation

Rework the function and means of access to separate 'notes' and 'notebook' levels of navigation.

Iteration 2

For this version, I removed the bottom navigation and merged the note and list functionality into a single navigation screen. The dump takes up a portion of the bottom half of the screen, and the notebook navigation takes up the rest.

Prototype

Low-fidelity Prototype (Version 1)

Above is the first version of the low-fidelity prototype. All assets were made by me with the exception of the iPhone keyboard asset.

Goals for this version:
  1. Demonstrate the layout and general navigation through the app
  2. Show how the "Dump" feature fits in
  3. Show how the text toolbar will work

Home Screen

The home screen contains 4 major sections:

  • Search bar - Matches typed text and finds all notebooks and notes
  • Notebooks section - Horizontal scroll of icons with notebook names
  • Notes section - Displays notes and subnotes within the selected notebook
  • The Dump - A readily-available, open note for quick "dumping" of ideas

The Dump & Text Tools

The user can open the dump from the home screen and see it full-screen. The text toolbar provides comprehensive formatting options:

  • Undo - Reverses changes
  • Image insertion - Add images to notes
  • Bold / Italic - Text formatting
  • Font toggle - Switch between sans-serif and serif
  • List toggle - Convert block to a list item that can be ticked off
  • Reminder - Add time-based reminders to blocks
  • Alignment - Left, center, right, or justify
  • Move - Reorder blocks up or down

List toggle - Converts the block to a list item that can be ticked and crossed off

Reminder toggle - Set date and time for reminders. Icon appears in text space showing the block has a reminder

Clear dump - Tap the meatball menu to clear the dump or move content to a note

Dump Expand/Contract Button

As a final update, I added a button to expand/contract the dump and removed the button to go to the 'Notebooks' screen. This creates a more intuitive full-screen mode and helps convey that the app's full functionality is within one main screen.

Test 2

A more structured user test to examine how users interact with the existing flows.

Aims

  • See if users can understand what the app is and does
  • Compare what users would do with how the app is actually structured
  • Find areas where users are confused or frustrated
  • See what users think buttons with no functionality will do

Methodology

8-step structured test including:

  • Pre-test setup and warnings
  • Home screen tasks and observations
  • Dump interaction tasks
  • Post-test questions about overall impressions
